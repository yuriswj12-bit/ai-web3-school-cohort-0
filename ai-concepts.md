# AI 基础概念笔记

这些概念用我自己的话整理。每个包含一句话解释、一个具体例子和一个常见误区或使用边界。目的是建立 Agent、workflow、AI coding 和工具调用的共同语言。

我基于训练数据和实际使用经验重新组织表述。没有直接复制外部输出。每个定义都聚焦实践边界。

## LLM

LLM 是大型语言模型。它通过海量文本训练来预测下一个词并生成连贯输出。

例子: 输入“分析当前 SOL 市场情绪”，它能返回一段带理由的总结。

误区: 很多人以为 LLM 具备实时事实或绝对准确。它只是概率模型。超出训练数据范围时容易生成看似合理但错误的答案。

## Prompt

Prompt 是给模型的指令文本。它决定模型如何理解任务并控制输出方向。

例子: 用“用最简短的句子解释工具调用”作为 prompt，模型直接返回核心要点而非长文。

误区: 认为 prompt 越长越详细越好。实际中模糊、矛盾或过长的指令会让模型偏离重点，输出质量下降。

## Context Window

Context window 是模型一次能处理的最大文本长度。它包括输入和生成内容。

例子: 一个 128k token 的 window 能装入整篇长文档的总结任务。超出后早期信息被截断。

误区: 以为模型能记住无限历史。实际超出 window 后前面关键细节直接丢失，导致后续回答断裂或重复。

## Workflow

Workflow 是把多个步骤组织成固定流程的结构。它定义任务分解、执行顺序和输出传递。

例子: 先用 LLM 提取新闻要点，再调用 Agent 分析情绪，最后生成交易信号。

误区: 把随意多轮聊天当成 workflow。没有明确结构就难以复现、调试和持续优化。

## Agent

Agent 是能自主规划和执行任务的系统。它结合 LLM、工具调用和循环决策来达成目标。

例子: 一个交易 Agent 先查实时价格、判断买入信号、再调用下单工具完成操作。

误区: 把单次 prompt 调用当成 Agent。真正的 Agent 需要状态保持、错误处理和多步循环。缺少这些只是普通调用。

## Tool Use

Tool use 是让模型调用外部函数或 API 的能力。它让模型获取实时数据或执行实际动作。

例子: 模型调用 OKX DEX API 查询当前代币价格，而不是靠旧记忆回答。

误区: 认为模型天生能访问互联网或直接操作系统。必须先明确注册工具、定义参数和权限才能安全调用。

## AI Coding

AI coding 是使用 AI 辅助编写、调试和优化代码的过程。它结合 prompt engineering 和工具调用来加速开发。

例子: 用 AI 生成单元测试框架，然后人工 review 并补充边界条件。

误区: 以为 AI 能写出完美无 bug 的生产代码。实际生成代码经常包含隐藏错误或不安全模式，必须配合 guardrails 和人工检查。

## Guardrails

Guardrails 是防止模型输出有害或越界内容的机制。它通过规则、过滤器或验证层限制响应范围。

例子: 在交易场景中阻止模型生成“保证盈利”的建议或非法操作指令。

误区: 认为 guardrails 能完全消除风险。高级 prompt 仍可能绕过简单规则。生产环境需要多层验证。

## Tracing

Tracing 是记录 Agent 或 workflow 执行过程的日志系统。它帮助调试、审计和优化决策路径。

例子: 查看每次 tool call 的输入输出、LLM 思考过程和最终决策，便于复现问题。

误区: 忽略 tracing 导致生产问题难以定位。关键流程必须开启完整日志，否则优化无从谈起。

## Human-in-the-Loop

Human-in-the-loop 中文常称人在回路。它是在关键步骤引入人工审核的流程。它平衡自动化效率和安全控制。

例子: AI 生成交易信号后，由交易员确认风险和仓位大小再执行下单。

误区: 以为全自动流程最好。忽略人工审核在高风险或高价值场景会造成不可逆损失。
